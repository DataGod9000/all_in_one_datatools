DataTools Portfolio — Overview & Capabilities

A detailed introduction to the DataTools Portfolio application: what it is, what problems it solves, and a full account of its functions and strengths.

1. Introduction

DataTools Portfolio is an internal data platform for managing database tables, schema changes, data quality checks, and table comparisons in a governed, audit-friendly way. It is built SQL-first and runs against Supabase Postgres (or any PostgreSQL-compatible database), with a FastAPI backend and a React (Vite) frontend.

The application provides:

  • DDL lifecycle — Parse, validate, and apply CREATE TABLE statements with governance rules (e.g. naming, required column comments).
  • Table comparison — Compare two tables (e.g. dev vs prod) by join keys and optional partition filters, with row counts, missing rows, and column-level value differences.
  • Data validation — Run quality checks (total rows, null counts per column, duplicate rows) and store results for review.
  • Asset management — Browse tables, backups, and scheduled deletions; view DDL and column details; schedule safe deletes and restore from backup.
  • Approval workflow — For production (PROD), table creation, deletion, and restore go through a two-step approval (Team Lead → Data Governance) before execution.
  • Ad-hoc query — Run single SELECT or INSERT statements (with limits and safety checks) for quick exploration.

All write actions are audited in datatools.audit_log, and the system enforces identifier validation and allowed schemas so that no arbitrary user SQL is executed.

2. Problems It Solves

2.1 Inconsistent or unsafe schema changes

Teams often apply DDL manually or via ad-hoc scripts, leading to typos, non-standard names, and risky statements (e.g. DROP, TRUNCATE). DataTools Portfolio:

  • Parses and validates CREATE TABLE only; rejects DROP, ALTER, TRUNCATE, COPY, GRANT, REVOKE.
  • Enforces a table naming governance pattern (e.g. {layer}_josephco_{domain}_{tablename}_{granularity}) so tables are discoverable and consistent.
  • Requires English and Chinese column comments for data governance before apply.

2.2 No visibility into dev vs prod differences

Comparing tables across environments (e.g. dev and prod) is tedious and error-prone when done manually. The app:

  • Runs comparisons in the database (SQL joins), so large tables are not loaded into memory.
  • Supports join keys and optional partition (e.g. pt) filters, with suggested join keys from column name/type matching.
  • Stores run history (row counts, missing-in-left/right, column diffs, samples) so teams can review and re-run.

2.3 Ad-hoc data quality checks with no history

Data quality (nulls, duplicates, row counts) is often checked one-off in SQL with no record. The app:

  • Runs validation jobs (total rows, null counts per column, duplicate rows) in the background.
  • Persists results in datatools.validation_runs so teams can track quality over time and share links to runs.

2.4 Production changes without approval or audit

Production table creation, deletion, or restore can cause outages or data loss if done without review. The app:

  • Creates tables in DEV immediately; in PROD, create/delete/restore go through table requests.
  • Uses a two-step approval (Team Lead → Data Governance) for PROD requests.
  • Logs every write action in datatools.audit_log with action, schema, and details.

2.5 Risky or untracked table deletion

Deleting tables without backup or a grace period is dangerous. The app:

  • Schedule delete: clones the table to back_up_<name>_YYYYMMDD, renames the live table to to_be_deleted_<name>, and schedules physical deletion after 7 days.
  • Restore backup: renames a backup table back to the original and cleans up the deletion schedule.
  • In PROD, delete and restore require approval before execution.

2.6 Scattered table metadata and DDL

Finding table definitions and column info often requires digging in the database. The app:

  • Lists tables (from registry), backups (back_up_%), and to-be-deleted tables with search.
  • Exposes table details (row count, size), columns, and DDL via API and UI for copy/paste and documentation.

3. Functions and Capabilities (Detailed)

3.1 DDL — Parse, suggest, apply

Function / EndpointDescription
POST /ddl/parse Parses a CREATE TABLE DDL string and returns schema, table name, columns (name, type, nullable, default), and constraints. No database access. Validates that only CREATE TABLE is present.
POST /ddl/suggest-column-comments Uses OpenAI (GPT-4o-mini) to generate short English and Chinese column comments from column names and types. Requires OPENAI_API_KEY in .env. Returns a list of column_name, comment_en, comment_zh.
POST /ddl/suggest-table-name Uses AI to suggest a single table name following the governance pattern (e.g. ods_josephco_growth_sample_users_di). Validates the suggestion against the same regex; returns error if not compliant.
POST /ddl/apply Validates env_schema (dev/prod), parses DDL, enforces governance (table name + column comments), builds and executes CREATE TABLE in the database, applies column comments, upserts table_registry, and writes to audit_log. DEV only for direct apply; PROD create goes through approval.

Why it’s valuable: Single place to edit DDL, get AI-assisted comments and naming, and apply safely with full audit and governance.

3.2 Compare — Suggest keys, run comparison

Function / EndpointDescription
POST /compare/suggest-keys Given left/right table names and schema(s), fetches column names and types from both tables and suggests join key candidates (e.g. same name + same type). Returns scored candidates (e.g. by name similarity and type match) up to max_candidates.
POST /compare/run Queues a comparison job: left vs right table (optionally different env schemas), join keys (or key pairs), optional compare columns, optional partition filters (left_pt, right_pt). Runs in a background thread. Returns run_id and status pending. Job computes: left/right row counts, missing_in_right, missing_in_left, sample of unjoined rows, and per-column value difference counts (for chosen compare columns). Results stored in datatools.compare_runs.
GET /compare/runs Lists comparison runs with optional env_schema filter and limit. Returns id, tables, schemas, join keys, compare columns, result_json, status, error_message, created_at.
GET /compare/runs/{run_id} Returns a single run (for polling status and viewing results).

Why it’s valuable: Compare dev vs prod (or any two tables) without writing custom SQL; history and samples are stored for debugging and reporting.

3.3 Validate — Run and list

Function / EndpointDescription
POST /validate/run Queues a validation job for a given target_table and env_schema. Runs in background: total row count, null count per column, duplicate row count (full-row duplicates). Writes result to datatools.validation_runs and audit_log.
GET /validate/runs Lists validation runs with optional env_schema and limit.
GET /validate/runs/{run_id} Returns one run (status, result_json, error_message).

Why it’s valuable: Consistent, repeatable data quality checks with stored results and no one-off scripts.

3.4 Assets — Tables, backups, DDL, schedule delete, restore

Function / EndpointDescription
GET /assets/tables Lists tables: filter=tables (from table_registry), backups (names like back_up_%), or to_be_deleted (names like to_be_deleted_%). Optional env_schema and search q. Excludes backups and to_be_deleted from “tables” view when appropriate.
GET /assets/table-columns Returns column list (name, type) for a given schema and table from information_schema.
GET /assets/table-details Returns row count and approximate size for a table (and optional columns).
GET /assets/table-ddl Builds and returns CREATE TABLE DDL for the table (from registry or from database metadata). Useful for documentation and copy/paste.
POST /assets/schedule-delete Creates backup back_up_<name>_YYYYMMDD, renames table to to_be_deleted_<name>, inserts into deletion_schedule (delete_after = now + 7 days), removes from table_registry. Audit logged.
POST /assets/restore-backup Restores a backup table: drops to_be_deleted_<name> if present, renames backup to original name, cleans deletion_schedule, adds/updates table_registry. Audit logged.

Why it’s valuable: One place to see all assets, get DDL, and perform safe delete/restore with backups and scheduling.

3.5 Query

Function / EndpointDescription
POST /query/run Executes a single SELECT or INSERT statement. SELECT: automatically adds LIMIT 500 if not present; returns columns and rows. INSERT: executes and returns rows_affected. Comments stripped for validation; only one statement allowed. No DROP/ALTER/TRUNCATE etc.

Why it’s valuable: Quick ad-hoc exploration and small inserts without opening a separate DB client.

3.6 Approval workflow — Table requests

Function / EndpointDescription
POST /api/table-requests Creates a table request. DEV: create is applied immediately (DDL apply + column comments + table_registry + created_tables). PROD: create/delete/restore create a request with status pending_approval; no execution until approved. Delete/restore only allowed for PROD.
GET /api/table-requests Lists requests; optional status filter (pending_approval, pending_governance, approved, rejected). pending_approval includes pending_governance.
PATCH /api/table-requests/{id}/approve Approval step. step=team_lead: pending_approval → pending_governance (team lead approval). step=governance: pending_governance → approved and then executes the action: for create, records in created_tables; for delete, executes schedule-delete (backup + rename + deletion_schedule); for restore, executes restore-backup. All with audit.
PATCH /api/table-requests/{id}/reject Rejects the request with optional rejection_reason.
GET /api/created-tables Lists tables created through the platform (created_tables) for history and traceability.

Why it’s valuable: PROD changes are gated by a clear two-step approval and fully audited; DEV remains fast for iteration.

3.7 Health and frontend

Function / EndpointDescription
GET /health Returns {"status": "ok"} for liveness checks.
GET /{path} Serves the React SPA: static files when they exist, otherwise index.html for client-side routing. API docs at /docs.

4. Why This App Is Valuable (The “Greatness”)

  • SQL-first, no arbitrary execution: All database work is controlled (parsed DDL, fixed comparison/validation queries, approved requests). Identifiers are validated; only allowed schemas (e.g. dev, prod) are used. This reduces risk of injection and accidental damage.
  • Governance by design: Table naming and dual-language column comments are enforced before tables are created, improving consistency and documentation.
  • Full audit trail: Every write (DDL apply, compare run, validate run, schedule delete, restore, approval) is logged in datatools.audit_log with action, env_schema, and details for compliance and debugging.
  • Safe deletion and restore: Schedule delete always creates a dated backup and a 7-day grace period; restore is explicit and tracked. In PROD, both require approval.
  • Scalable compare and validate: Comparisons and validations run in the database (JOINs, COUNTs, DISTINCT) and in background threads, so large tables don’t block the API and results are stored for later review.
  • Clear PROD vs DEV: DEV allows direct create; PROD requires approval with a two-step workflow (Team Lead → Data Governance), so production stays protected while development stays agile.
  • Unified UX: One web app for DDL, compare, validate, assets, approval center, request history, and run query reduces context-switching and standardizes workflows.
  • AI-assisted productivity: Optional AI suggestions for column comments and table names (with governance validation) speed up authoring without sacrificing standards.

5. Technical Summary

Backend: Python 3.10+, FastAPI, psycopg (Postgres), sqlglot (DDL parse), OpenAI (optional).

Frontend: React, Vite, React Router; theme (light/dark), toasts, responsive layout.

Database: Supabase Postgres (or compatible). Schema datatools with tables: audit_log, table_registry, compare_runs, validation_runs, deletion_schedule, table_requests, created_tables. Setup script: scripts/setup_datatools_schema.sql.

Run: cd frontend && npm run build then python main.py or uvicorn main:app --port 8000. App at http://localhost:8000, API docs at http://localhost:8000/docs.
